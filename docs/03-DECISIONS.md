## ADR-001: 情感模型选用 PAD 而非 Big Five

- **日期**：2026-05-29
- **问题**：角色情感该用什么心理学模型？
- **决策**：采用 PAD (Pleasure-Arousal-Dominance) 三维模型
- **理由**：Big Five 是人格特质（稳定），PAD 是情感状态（动态），更适合追踪章节间的情感波动
- **后果**：情感坐标用 3 个浮点数，而非 5 个维度，降低计算复杂度
- **状态**：✅ 已冻结

## ADR-009: 采用 Tier 0-4 分级定价模型

- **日期**：2026-06-02
- **问题**：如何在 $20/月硬约束下最大化 AI 能力覆盖？
- **决策**：采用五级定价模型（Tier 0-4），按任务复杂度分级调用
  - Tier 0: 免费，纯规则引擎（实体匹配、数值比较、关键词检测）
  - Tier 1: ~$0.01/次，轻量 AI + 短 Prompt（单引擎增量分析）
  - Tier 2: ~$0.05/次，标准 AI（章节定稿全分析）
  - Tier 3: ~$0.2/次，全量模型 + 长上下文（剧情推演、角色模拟）
  - Tier 4: ~$1/次，后台异步（全书审查）
- **理由**：不区分定价会导致简单任务浪费 Token、复杂任务不够用；分级后本地 7B 处理 70% 常规任务，云端仅用于深度推理
- **后果**：编排器需要感知 Tier 并在预算不足时自动降级；降级策略从简单开关变为逐级关闭
- **状态**：✅ 已冻结

## ADR-010: 事件驱动调度 + 按需触发引擎

- **日期**：2026-06-02
- **问题**：8 个引擎如何高效协作而不浪费 API 调用？
- **决策**：采用事件驱动架构，用户操作仅触发相关引擎
  - 修改对话 → 角色引擎 + 文辞引擎
  - 修改设定 → 世界引擎 + 级联修正引擎
  - 章节定稿 → 全引擎并行（非串行）
  - 全书审查 → 后台异步任务
- **理由**：全量串行分析成本高、延迟大；事件驱动只调用相关引擎，响应快且省 Token
- **后果**：编排器需要维护事件-引擎映射表和并行调度逻辑；Phase 1 映射表仅含 2 引擎
- **状态**：✅ 已冻结（ADR-006 的正式化扩展）

## ADR-011: Phase 1 MVP 仅激活 2 个引擎

- **日期**：2026-06-02
- **问题**：Phase 1 应该包含多少引擎？
- **决策**：仅激活角色引擎 + 世界引擎。其余 6 个引擎（叙事/文辞/主题/经济/读者预期/构思）全部标记 Phase 2+

- **后果**：MVP 代码量从预计 8-10 万行降至 3-5 万行；一致性守卫 Phase 1 仅检查 3 维（角色/世界/空间）；级联修正器仅做关键词扫描
- **状态**：✅ 已冻结（ADR-007 的进一步裁剪）

## ADR-012: 本地 Qwen2.5-7B 处理 70% 常规任务

- **日期**：2026-06-02
- **问题**：如何在 $20/月预算下覆盖足够多的调用次数？
- **决策**：常规任务（语料切片 embedding、简单分类、关键词提取）使用本地 Qwen2.5-7B；云端 API（MiMo/DeepSeek）仅用于深度推理（行为预测、规则冲突分析）
- **理由**：本地模型零 API 成本，7B 级别足以处理分类和 embedding 任务；云端模型保留给需要强推理能力的场景
- **后果**：需要维护本地模型部署和版本；embedding 质量略低于 BGE 专用模型；用户机器需支持 7B 模型推理（建议 16GB+ RAM）
- **状态**：✅ 已冻结

## ADR-013: 语料切片标准 — 500 字场景级

- **日期**：2026-06-02
- **问题**：语料切片的粒度和标准是什么？
- **决策**：按场景（Scene）切片，每片约 500 字，标注 `PAD状态 + 行为模式 + 决策后果 + 互文意象`。生产流水线：AI 粗提取（本地 7B）→ 人工精校验（只审不标，2-3 分钟/片）→ 反馈修正
- **理由**：段落级切片丢失上下文，章节级切片太粗无法精确检索。500 字场景级是最小可独立理解的行为单元，且人工校验成本可控
- **后果**：Phase 1 启动集从 100 片降为 20 片（红楼 10 + 自有 10）；切片标注格式需冻结为 JSON Schema
- **状态**：✅ 已冻结
- **变更记录**：
  - 2026-06-08: 放宽字数限制为 450-800 字，场景完整性优先于字数限制。原因：人工审核已完成，500 字标准的"人工校验成本可控"目的已达成

## ADR-015: Phase 1 LLM 集成 — 使用 MiMo 云 API 替代本地模型

- **日期**：2026-06-09
- **问题**：Phase 1 M3 声称"引擎核心完成"，但实际上角色引擎和世界引擎的"智能分析"全部是 `if-else` 关键词匹配。行为预测是 `"笑"→"微笑/欢笑"`，PAD 情感是读取 metadata 标签，世界规则校验是关键词集合匹配。所有测试结果显示 `needs_human_review=True`，置信度 < 0.5。
- **决策**：
  1. 使用小米 MiMo TokenPlan API（`mimo-v2.5-pro`）作为 LLM 后端，替代原计划的本地 Qwen2.5-7B（ADR-012 已修订）
  2. 新增 `src/llm/` 模块：config、client、prompts、cost_tracker
  3. 每个 LLM 调用点保留 fallback 链：LLM → 关键词匹配 → 默认值，确保无 API key 时仍可运行
  4. 引入语料自扩充回路（Enricher）：LLM 高置信度（>0.6）分析结果自动存储为动态切片，供后续检索使用
  5. 文本分析长度：PAD 分析 3000 字，行为预测 4000 字，事件描述 1500 字（原 500-800 字严重不足）
- **理由**：
  - 本地部署 Qwen2.5-7B 需要 16GB+ RAM 和模型服务基础设施，超出 Phase 1 工程能力
  - MiMo TokenPlan 提供免费额度 + OpenAI 兼容接口，零基础设施成本
  - 关键词匹配引擎无法通过 M6 验证标准（行为预测与语料模式相关性 > 0.6）
  - 云 API 集成只需 `openai` SDK（1 个依赖），与已允许的 Flask 例外一致
- **后果**：
  - API key 存储在 `config/llm.json`（gitignored）
  - 月度预算追踪器（CostTracker）实现三层熔断：50%/$10 警告、80%/$16 限制、95%/$19 阻断
  - 每次分析成本约 $0.003-0.007（取决于文本长度）
  - 需要更新 PyInstaller spec 打包 `src/llm/` 和 `openai` 依赖
  - 原 ADR-012（本地 7B）标记为已修订
- **状态**：✅ 已冻结

## ADR-016: 语料自扩充回路（Enricher）

- **日期**：2026-06-09
- **问题**：20 个静态语料切片 + TF-IDF 检索的相似度在 0.12-0.28 之间，接近随机。系统需要更多高质量标注数据来提升检索质量。
- **决策**：
  1. 每次 LLM 分析产生高置信度（>0.6）结果时，自动创建"扩充切片"
  2. 扩充切片融合原文 + LLM 分析文本（行为描述、PAD 坐标），存储在 `projects/<id>/enriched/`
  3. Retriever 改为双索引搜索：静态语料库 + 动态扩充索引
  4. 动态索引上限 100 片，超出后移除最旧的
  5. LLM 精排：TF-IDF 取 top-20，LLM 从候选中选最佳 top-3
- **理由**：TF-IDF 基于词频，通过将 LLM 的语义理解（行为描述文本）融入切片正文，简单的词频匹配也能捕捉深层语义关联。使用越多，检索越准。
- **后果**：扩充切片按项目隔离（不污染公共语料），每次分析后自动触发索引重建
- **状态**：✅ 已冻结

### ADR-016: Phase 2 叙事引擎架构

- **日期**：2026-06-10
- **决策**：叙事引擎作为 Phase 2 首个新增引擎，提供伏笔追踪、因果链构建和叙事问题检测三大能力。采用与 Phase 1 引擎相同的架构模式：LLM 优先 + 关键词 fallback，不直接写记忆，由编排器统一调度。
- **架构要点**：
  1. 伏笔有完整生命周期（planted → partial → resolved），由引擎内部注册表管理
  2. 因果链提取基于事件间逻辑关系，非简单时间顺序
  3. 跨章节伏笔回收检查支持 LLM 语义判断和关键词 fallback 两种模式
  4. 叙事问题检测覆盖四类：因果循环、长期遗忘、因果断层、类型重复
  5. 新增 4 个 LLM 任务类型（foreshadow_detect / causal_extract / resolution_check / event_predict）
  6. 编排器新增 "narrative" 和 "full_analyze" 两种路由动作
- **理由**：伏笔管理是长篇网文创作的核心痛点，因果链是叙事逻辑的基础设施。Phase 1 的角色引擎和世界引擎解决了"谁做了什么"和"是否符合规则"，叙事引擎解决"为什么发生"和"之前暗示了什么"。
- **后果**：新增 `src/engines/narrative.py`（~370 行），新增 4 个 LLM system prompt，Tier 成本表新增 4 个任务类型（单次调用成本 ~$0.001），一致性守卫维度扩展至 foreshadowing / causal / narrative_consistency
- **状态**：✅ 已冻结
